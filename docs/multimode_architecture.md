# Многослойная Архитектура Aventura

Проект должен поддерживать несколько игровых режимов, которые используют одну базу героев, но имеют отдельную механику и отдельные таблицы.

## Общий Центр

Единый источник истины для игроков и героев:

- `players`;
- `characters`;
- `change_log`;
- инвентарь, дублоны, характеристики, уровни, заклинания и помощники внутри `characters`.

Все режимы игры ищут персонажа по Telegram ID через общие сервисы.

## Режим 1: PvE Миссии

Текущий основной режим.

Таблицы:

- `turns`;
- `missions`;
- `mission_participants`;
- `actions`;
- `results`;
- `city_chronicle`.

Бот:

- `aventura_bot.bot`.

Отвечает за ежедневные ходы, миссии, экспорт, импорт результатов и хронику города.

## Режим 2: PvP Арена

Будущий отдельный режим для дуэлей и турниров между игроками.

Заложенные таблицы:

- `arena_tournaments`;
- `arena_matches`.

Назначение:

- турниры;
- вызовы на дуэль;
- пары участников;
- заявки/действия бойцов;
- результат боя;
- награды или последствия через общий `change_log`.

Важно: PvP не должен использовать таблицы `missions` и `turns`.
У него свой цикл матчей, но герои берутся из общей таблицы `characters`.

Возможный будущий entrypoint:

```text
python -m aventura_bot.arena_bot
```

## Режим 3: Ферма Гильдии

Будущий отдельный режим про развитие инфраструктуры Авентуры и Каррок Манора.

Заложенные таблицы:

- `guild_farms`;
- `guild_farm_contributions`.

Назначение:

- ресурсы гильдии;
- здания и улучшения;
- проекты;
- вклады персонажей;
- развитие Каррок Манора;
- бонусы, открывающие новые возможности для PvE/PvP.

Важно: ферма не должна хранить отдельные копии героев.
Она ссылается на `characters`, когда персонаж вносит вклад или получает эффект.

Возможный будущий entrypoint:

```text
python -m aventura_bot.guild_farm_bot
```

## Общие Правила Для Всех Режимов

- Не создавать отдельные базы персонажей.
- Не дублировать механику инвентаря и дублонов.
- Все изменения героя проводить через общие сервисы или общий `change_log`.
- Режимы могут иметь свои таблицы состояния, но не должны править чужие таблицы без явной причины.
- SQLite подходит для MVP, но операции записи должны быть короткими и атомарными.

## Будущее Разделение Сервисов

Когда режимов станет больше, стоит вынести логику по файлам:

```text
aventura_bot/services/
  game.py          # общие персонажи, награды, уровни, инвентарь
  pve.py           # PvE-миссии и ходы
  trade.py         # обмен предметами и дублонами
  arena.py         # PvP
  guild_farm.py    # ферма гильдии
```

Сейчас часть логики еще живет в `game.py`; это нормально для MVP.
