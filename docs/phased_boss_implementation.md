# Phased Boss Implementation

Это рабочая спецификация для внедрения многофазовых боссов в игру Авентуры.

## Цель

Добавить особый тип миссии, где:

- одна сцена;
- один босс;
- 2-3 фазы;
- до 9 участников;
- между фазами меняется сам босс, а не место действия;
- вступивший герой не может выйти из боя до победы или поражения.

## Тип миссии

```yaml
type: boss
subtype: phased
```

## Базовые поля в turn.yaml

```yaml
missions:
  - title: "Имя босса"
    type: "boss"
    subtype: "phased"
    continuation_key: "raduzhny_stochny_koloss"
    phase: 1
    max_phase: 3
    difficulty: 28
    max_participants: 9
    party_locked: true
    lock_warning: "Вступив в бой, герой останется в нем до победы или поражения."
    boss_name: "Радужный Сточный Колосс"
    boss_theme: "живая алхимическая слизь, канализация, ядовитые испарения"
    description: |
      Описание текущей фазы...
    threat:
      type: "boss_aberration"
      danger: 5
      notes: "Служебные заметки о фазе."
```

## Смысл полей

- `type`: базовый тип миссии.
- `subtype`: конкретный режим внутри типа.
- `phase`: текущая фаза босса.
- `max_phase`: общее число фаз, только 2 или 3.
- `continuation_key`: постоянный ключ этой босс-арки между ходами.
- `difficulty`: сложность именно текущей фазы.
- `max_participants`: максимум участников, по умолчанию 9.
- `party_locked`: игроки не могут покинуть бой после вступления.
- `lock_warning`: текст для UI.
- `boss_name`: каноническое имя босса.
- `boss_theme`: короткая служебная тема для генерации описаний, фаз и трофеев.

## Правила UI

В карточке миссии показывать:

```text
Тип: босс-миссия
Фаза: 1/3
Вступив в бой, герой останется в нем до победы или поражения.
Участников: до 9
```

### Поведение интерфейса

- `/missions` должен показывать такую миссию как отдельную карточку.
- Если герой уже вступил в phased boss, он не может выбрать обычную миссию, пока бой не завершится.
- `/my_action` должен явно показывать:
  - что это phased boss;
  - текущую фазу;
  - что выход невозможен до завершения.

## Логика участия

- Вступивший герой закрепляется за этой миссией.
- Состав группы сохраняется на всю битву.
- Для сохранения состава между ходами используется `continuation_key`.
- Новые участники могут войти только пока миссия на текущем ходе открыта и пока не превышен лимит.
- Если phased boss уже идет в следующих ходах, состав сохраняется и свободного входа больше нет, если мастер не решит иначе отдельным override.

## Расчет фазы

Личный вклад героя:

```text
уровень героя
+ релевантная характеристика
+ 1 применимый предмет
+ 1 применимое заклинание
+ 1 применимый питомец/фамильяр
+ 1 применимый спутник
+ 1 применимый маунт
```

Общий вклад группы:

```text
сумма личных вкладов всех участников
```

### Провал хода игрока

Если текст игрока:

- бессвязный;
- пустой;
- не относится к фазе;
- не помогает решать конфликт;

то его вклад в фазу:

```text
0
```

### Результат фазы

- если общий вклад группы >= difficulty текущей фазы:
  - фаза пройдена;
- если общий вклад группы < difficulty текущей фазы:
  - вся миссия получает `failed`.

## Переход между фазами

Если текущая фаза пройдена и это не последняя фаза:

- миссия получает `ongoing`;
- в следующем ходе публикуется та же миссия;
- `phase` увеличивается на 1;
- `difficulty` заменяется на сложность новой фазы;
- описание фазы переписывается под новое состояние босса.

### Важно

Меняется:

- состояние босса;
- его атаки;
- уязвимости;
- давление на арену;
- художественная картина боя.

Не меняется:

- сама арена как отдельная новая миссия;
- сам базовый конфликт;
- состав группы.

## Завершение миссии

- если пройдена последняя фаза -> `completed`
- если провалена любая фаза -> `failed`

## Награды по фазам

### Нефинальные фазы

За каждую успешно пройденную нефинальную фазу:

- всем участникам:
  - `level delta = 1`

Промежуточные награды:

- по желанию мастера или генератора;
- только небольшие;
- не обязательны;
- не заменяют финальную награду.

## Финальная награда

Если последняя фаза пройдена и миссия завершена:

### Лично успешный участник получает

- `level delta = 2`
- обычную награду по `reward_roll`
- отдельный `boss_trophy`

### Лично неуспешный участник получает

- `level delta = 1`
- только утешительное золото
- без `reward_roll`
- без `boss_trophy`

## Boss trophy

`boss_trophy` - особая награда победителя босса, отдельная от обычного `reward_roll`.

### Формат

Рекомендуется оформлять как уже существующий игровой объект:

- `inventory`
- `spells`
- `pet`
- `companion`
- `mount`
- иногда `status`

### Принцип

`boss_trophy` должен:

- быть связан с именем, темой и природой босса;
- звучать как полноценная игровая награда;
- не быть сырой частью тела;
- иметь уникальное имя.

### Уровень boss_trophy

```text
boss_trophy_min = floor(boss_difficulty * 0.5)
boss_trophy_max = floor(boss_difficulty * 0.7)
```

Где `boss_difficulty` - общая верхняя сложность босса, обычно сложность финальной фазы.

Пример:

- финальная сложность босса `55`
- `boss_trophy` уровня `27-38`

### Несколько победителей

Если лично успешных победителей несколько:

- каждый получает свой `boss_trophy`;
- трофеи должны быть разными по имени;
- все трофеи должны быть связаны с одним и тем же боссом.

## Правила именования boss_trophy

Примеры в инструкциях не образуют обязательный шаблон.

Нельзя, чтобы несколько боссов подряд:

- выглядели как вариации голема;
- имели одинаковый словарь наград;
- повторяли одни и те же шаблоны вроде "перчатка / броня / малый голем".

Перед генерацией boss trophy сначала определить:

- природу босса;
- его материал или среду;
- стиль давления на сцену;
- его уникальный словарь;

и только потом собирать названия наград.

### Хороший принцип

Награда должна быть:

- тематически из босса;
- но по форме нормальной вещью мира.

Примеры принципа:

- не "глаз колосса",
- а "Линза Радужного Осадка";
- не "кусок голема",
- а "Латная Перчатка Стального Марша".

## Провал phased boss

Если группа провалила любую фазу:

- миссия получает `failed`;
- уже полученные уровни за предыдущие успешно пройденные фазы сохраняются;
- можно выдать компенсацию по глубине прохождения;
- обычный `reward_roll` за финал не выдается;
- `boss_trophy` не выдается.

### Компенсация

Компенсацию удобнее всего выдавать золотом.

Базовый подход:

- провал на ранней фазе -> малая компенсация;
- провал на поздней фазе -> более заметная компенсация.

## Как phased boss должен выглядеть в export

Экспорт должен содержать обычные данные миссии плюс служебные поля:

```json
{
  "mission_id": 42,
  "title": "Радужный Сточный Колосс",
  "type": "boss",
  "subtype": "phased",
  "phase": 2,
  "max_phase": 3,
  "difficulty": 38,
  "party_locked": true,
  "boss_name": "Радужный Сточный Колосс",
  "boss_theme": "живая алхимическая слизь и канализационные испарения",
  "participants": []
}
```

## Как phased boss должен выглядеть в result

### Если фаза не последняя и пройдена

- `status = ongoing`
- у всех участников:
  - `level delta = 1`
- в `world_changes` и `public_overview` описывается переход босса к следующей фазе

### Если последняя фаза пройдена

- `status = completed`
- лично успешные:
  - `level delta = 2`
  - `reward_roll`
  - `boss_trophy`
- лично неуспешные:
  - `level delta = 1`
  - `consolation_reward`

### Если фаза провалена

- `status = failed`
- без финального `reward_roll`
- без `boss_trophy`

## Порядок реализации

### Шаг 1

Добавить в модель миссии поддержку полей:

- `type`
- `subtype`
- `phase`
- `max_phase`
- `max_participants`
- `party_locked`
- `lock_warning`
- `boss_name`
- `boss_theme`

### Шаг 2

На уровне UI:

- показать новые поля в карточке миссии;
- заблокировать выход и переключение с такой миссии;
- отразить фазу в `/my_action`.

### Шаг 3

На уровне генерации:

- научить `turn.yaml` создавать phased boss;
- не допускать повторяющегося архетипа боссов.

### Шаг 4

На уровне обработки:

- суммировать вклады группы;
- различать промежуточную и финальную фазу;
- выдавать `boss_trophy` только победителям финальной фазы.

### Шаг 5

На уровне result:

- поддержать `boss_trophy` как отдельную дополнительную награду победителя.
