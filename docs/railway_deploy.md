# Railway Deploy

Этот проект можно держать на Railway как постоянный Telegram-бот, а обработку ходов по-прежнему делать локально через Codex.

## Что будет жить в Railway

- сам бот 24/7;
- база персонажей и ходов;
- лавка, обмены, миссии, дедлайны;
- файлы, которые бот хранит у себя: импорт/экспорт, хроника, арты.

## Что останется у тебя локально

- генерация `turn.yaml`;
- обработка `export.json` в `result.json`;
- мастерские правки и новые правила.

Важно: после переезда в Railway у нас **не будет общей папки** между ботом и локальным проектом. Значит, рабочий цикл становится таким:

1. ты загружаешь `turn.yaml` боту;
2. бот сам ведет ход;
3. бот по дедлайну делает export;
4. ты скачиваешь export из Telegram;
5. локально в Codex делаешь `result.json`;
6. загружаешь `result.json` обратно боту.

## Почему Railway подходит

- запуск проще, чем на Oracle Cloud;
- бот можно держать как worker-процесс;
- можно подключить persistent volume;
- нам не нужен публичный веб-сайт.

## Что обязательно настроить в Railway

### 1. Новый проект

Создай новый проект из GitHub-репозитория с этим кодом.

### 2. Persistent Volume

Подключи volume и примонтируй его в:

```text
/data
```

Это важно: SQLite и рабочие файлы должны лежать не в эфемерной файловой системе контейнера.

### 3. Environment Variables

Добавь:

```text
TELEGRAM_BOT_TOKEN=...
ADMIN_TELEGRAM_IDS=123456789
DATA_ROOT=/data
```

Если админов несколько:

```text
ADMIN_TELEGRAM_IDS=123456789,987654321
```

### 4. Start command

Если Railway сам не подхватит `Dockerfile`, укажи команду запуска:

```text
python -m aventura_bot.bot
```

Но обычно с `Dockerfile` этого уже не требуется.

## Что хранится в volume

При `DATA_ROOT=/data` бот сам положит туда:

```text
/data/aventura.sqlite
/data/exports
/data/imports
/data/backups
/data/seeds
/data/workbench
/data/art
/data/chronicle
```

## Ограничение новой схемы

Раньше локальный бот и Codex делили одну папку, и это позволяло автоматом читать `data/exports/`.
После переезда в Railway этот автоконтур исчезает.

То есть для облачной версии нормальный мастерский режим такой:

- скачать export из Telegram;
- положить его в локальный проект;
- обработать через Codex;
- загрузить result обратно боту.

## Рекомендуемый порядок миграции

1. Сначала оставить локальную версию как резерв.
2. Поднять Railway-проект с новым ботом.
3. Проверить `/start`, `/profile`, `/shop`.
4. Создать тестовый ход.
5. Прогнать полный цикл: `turn.yaml -> export -> result.json -> publish`.
6. Только потом переводить игроков на новый бот.

## Если захотим сделать еще лучше потом

Следующий шаг после альфы — вынести SQLite в Postgres и разделить:

- bot runtime;
- game database;
- gm processing pipeline.

Но для альфы Railway + volume + SQLite — нормальный и быстрый вариант.
